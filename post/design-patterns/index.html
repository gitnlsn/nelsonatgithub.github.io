Design Patterns: Elements of Reusable Object-Oriented Software

One thing expert designers know not to do is solve every problem from first principles. Rather, they reuse solutions that have worked for them in the past. When they find a good solution, they use it again and again. Such experience is part of what makes them experts. Consequently, you’ll find recurring patterns of classes and communicating objects in many object-oriented systems. These patterns solve specific design problems and make object-oriented designs more flexible, elegant, and ultimately reusable. [1]

Presentation

I remember when I first took the book and read the Contents section. I decided not to read it entirely. I would just read Chapters 1, 2 and 6. Introduction, a Case Study and Conclusion. Chapters 4, 5 and 6 are a kind of catalog. I would memorize three patterns I’d find the most interesting. Later I would query the book for new patterns, as soon as I find necessary. The more I implemented things and read documentation, the more I would discover new patterns and would be forced to query the book. So it happened when I learned Android and Node Express. In these frameworks, I could perceive the same design patterns from the book, with absolutely awesome implementation.

I’ll present a definition to design patterns. And I’ll present occurrences of some patterns in frameworks, say: Observable Pattern, Chain of Responsibility and Builder Pattern.

I hope you enjoy reading.

This post is intended to beginner developers;

Definition

Classes or structures within a specific set of dependencies working well together to solve a specific and recurring problem.

In the book, the authors talk about the necessity of not repeating code, refactoring, the search for the right granularity. They discuss the experience they have designing and implementing software. Some solutions and slices of software design tended to repeat and always succeed. These were presented as design patterns.

The design patterns are described by an intent, a solution, and some consequences.

The intent present the problem the development deals with.

The solution present the proposed design that handles the intent.

The consequences are usually solving the problem, like loose coupling and versatility, but possible draw backs.

Observable Pattern

Define a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically. [1]

In Android, different parts of the user interface may require the same data. Let’s say several Fragments requesting the same value from a LiveData in a ViewModel. In the framework it was chosen the View components to be updated through Observable Pattern.

It is not a good design to force the UI to access the data and update itself. As events are asynchronous, a wrong execution order could result in outdated components.

  • Component 1 updates itself;
  • Source data is changed;
  • Component 2 updates itself;
  • Component 2 is now updated, but Component 1 is outdated.

In Observable Pattern, the observable object registers a list of observers. The observers define what to do with the new value. As soon as the observable object is updated, all observers are called together and it guarantees all of them are updated with the same data. As consequence, observable and observers have loose coupling, but they share forced updates.

  • Component 1 observes Source data and sets a handler;
  • Component 2 observes Source data and sets a handler;
  • Source data is changed;
  • Handlers of Components 1 and 2 are called;

That data flow dynamics is called model-view paradigm.

MutableLiveData<Object> observedData = new MutableLiveData();

// Two listeners to the observable data, with different handles
observedData.observe(context, newValue -> { handle1(newValue); });
observedData.observe(context, newValue -> { handle2(newValue); });

// An update to observed data should force execution of both handle1 and handle2 methods
observedData.set(new Object());

Chain of Responsibility

Avoid coupling the sender of a request to its receiver by giving more than one object a chance to handle the request. Chain the receiving objects and pass the request along the chain until an object handles it. [1]

The routing system of Express Framework must handle requests which differ in path, http headers, body, cookies. These different requests require different responses. Chain of responsibility is well suited in this case.

The chain of responsibility is composed of a list of handlers which may handle an arbitrary request. Each handler decides if it can or can not handle the request and if it allows the remaining handlers do so. So a handler for each possible request can be designed one by one. All requests can be forwarded to the chain, expecting a proper handler to exist.

The pattern avoids direct coupling between request and handler and improves scalability.

A switch-case can implement a chain of responsibility, if the request is a primitive depending on the language. In a more general case, a proper implementation and DTO is necessary.

const app = express();

// If the express server is requested the "/path-a", it returns "Hello, path A" string
app.use('/path-a', (req, res, next) -> {
   return res.send('Hello, path A');
});

// If the express server is requested the "/path-b", it prints "Hello, middleware B" 
//    and passes controll to the next handler, if it exists.
app.use('/path-b', (req, res, next) -> {
   console.log('Hello, middleware B');
   return next();
});

// If the express server is requested the "/path-b", it returns "Hello, path B" string
app.use('/path-b', (req, res, next) -> {
   return res.send('Hello, path B');
});

Builder Pattern

Separate the construction of a complex object from its representation so that the same construction process can create different representations. [1]

This is the first pattern I implemented. The Builder implementation seems like bits of repeating code. But on the outer class perspective, object creation is more organized and versatile. Also, it is more readable, once it interacts with a proper creation class.

// Builder pattern
public class MyClass {
   private Object a;
   private Object b;

   public Object getA() { return this.a; }
   public Object getB() { return this.b; }

   public static class Builder {
      private Object a;
      private Object b;

      public Builder setA(Object object) {
         this.a = object;
         return this;
      }

      public Builder setB(Object object) {
         this.b = object;
         return this;
      }
      
      public MyClass build() {
         MyClass instance = new MyClass();
         instance.setA(this.a);
         instance.setB(this.b);
         return instance;
      }
   }
}

// Class that instatiates MyClass
class AnotherClass {
   public static void foo() {
      /* ... */
      MyClass instance = new MyClass.Builder()
         .setA(new Object())
         .setB(new Object())
         .build();
      /* ... */
   }
}

Final Considerations

What I’ve learned from Design Patterns is a repertoire to quick start some non-trivial design development. If the program requires instantiation of an object, I’d ask if the instantiation repeats and how it repeats. If the program requires some specific behaviour, which behaviour is it? Does it follow a pattern? I’d query the book. Frameworks usually have those patterns implemented.

Beyond these Patterns, there are good software design principles, such as Single Responsibility, Interface Segregation, Dependency Inversion. Those principles are meant to provide scalability, portability, maintainability, efficiency. Many years designing software led to many refactoring, failures, successes and knowledge.
